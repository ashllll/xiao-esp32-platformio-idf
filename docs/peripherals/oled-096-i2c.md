# 0.96 寸 128×64 I²C OLED 显示屏

## 先识别控制器和模块

“0.96 寸、IIC、128×64”只是外形与接口描述，不是完整型号。常见四针模块使用 SSD1306，也存在 SH1106 或其他兼容程度不同的控制器；它们的显示 RAM 宽度、列偏移、初始化序列和驱动选择并不完全相同。网页商品标题、PCB 颜色或 `0x3C` 地址都不能单独证明控制器型号。

接入前记录：PCB 正反面照片、引脚顺序、控制器/面板料号、7 位地址、供电标称范围和是否有板载稳压/电平转换。没有这些证据时，本文采用最保守的 3.3 V I²C 接法，并把 SSD1306 作为待验证假设。

## 接线

| OLED 丝印 | XIAO | GPIO（C3 / S3 / C6） | 说明 |
|---|---|---:|---|
| `VCC` | `3V3` | — | 未核对模块原理图前只用 3.3 V |
| `GND` | `GND` | — | 共地 |
| `SDA` | D4 | 6 / 5 / 22 | `XIAO_I2C_SDA` |
| `SCL`/`SCK` | D5 | 7 / 6 / 23 | `XIAO_I2C_SCL`；四针 I²C 板的 `SCK` 通常指 SCL |

先确认模块丝印顺序；市售四针板的 `GND/VCC/SCL/SDA` 排列并不统一。ESP32 GPIO 不耐受 5 V，必须确认 SDA/SCL 上拉到 3.3 V。共享 ENS160、SGP41、SHT40 时还要核算并联上拉。

## 地址与控制器识别

- SSD1306 的 I²C 7 位从地址由 SA0 决定，可为 `0x3C` 或 `0x3D`；通用模块常见 `0x3C`，但应以扫描结果为准。
- 仅凭 ACK 无法可靠地区分 SSD1306 与相近控制器，因为很多模块使用同一地址且没有可读取的芯片 ID。
- SSD1306 内部是 128 列；SH1106 的显示 RAM 通常为 132×64。选错驱动时可能出现水平偏移、边缘噪点或整屏异常，不能用随意加列偏移掩盖未识别的硬件。

可靠做法是核对供应商 BOM/原理图或面板 FPC 标识，再用对应数据表初始化序列验证。将识别结果写进项目配置，而不是运行时无依据地反复尝试多套命令。

## ESP-IDF 集成

SSD1306 的 I²C 事务包含控制字节：

- `0x00`：后续字节为命令流。
- `0x40`：后续字节为显示数据。

128×64 单色 framebuffer 需要 `128 × 64 / 8 = 1024` 字节 RAM。驱动至少应提供：复位/初始化、开关显示、设置寻址范围、整帧或脏区域刷新、亮度/对比度控制和总线错误返回。初始化参数与面板接法有关，例如 COM 引脚配置、电荷泵和扫描方向，不能脱离具体模块照抄一组“万能魔法数字”。

本工程是纯 ESP-IDF：

- 总线层使用 `driver/i2c_master.h`，组件依赖 `esp_driver_i2c`。
- 显示刷新不要长期占用共享 I²C 总线；更新 UI 时优先合并写入，并为传感器 1 Hz 采样保留调度空间。
- 任何 ESP Registry 或 GitHub 显示驱动都要固定版本、检查许可证，并确认明确支持 ESP-IDF 6 和实际控制器。
- 不要把 Arduino 的 `Wire`、Adafruit_GFX 或 U8g2 示例误写成本项目可直接编译的依赖。

## 最小验证

1. 断电检查 VCC/GND 排列和 I²C 上拉电压。
2. 以 100 kHz 扫描，记录 `0x3C` 或 `0x3D`。
3. 显示全黑、全白、单像素边框、四角标记和逐列图案。
4. 检查边缘是否偏移、是否缺列，以及 128×64 像素是否全部可寻址。
5. 与所有传感器并联后连续运行，确认刷新期间没有传感器 NACK、CRC 错误或采样节拍抖动。
6. 记录最终控制器证据、初始化配置、地址和实测供电电流。

OLED 长时间显示静态高亮内容可能产生不均匀老化。实际 UI 应降低无必要的常亮强度并适时改变或关闭静态画面。
